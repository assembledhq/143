143 runs coding-agent work as a team workflow. A request becomes a visible session, the agent works in a sandbox, reviewers inspect the diff and preview, and GitHub stays the source of truth for review, CI, and merge rules.

## Good first path

- [Connect GitHub](/docs/getting-started/connect-github): give 143 access to the repositories it can work in.
- [Start a session](/docs/getting-started/start-a-session): choose a repo, branch, coding agent, and clear task.
- [Review and ship](/docs/getting-started/review-and-ship): inspect the transcript, diff, validation output, preview, and PR before merging.

## Build toward production use

- [Repo config](/docs/guides/repo-config): define setup, validation, and preview behavior in `.143/config.json`.
- [Preview environments](/docs/guides/previews): run live UI and full-stack previews from a session.
- [Linear agent](/docs/guides/linear-agent): assign or mention `@143` in Linear and keep progress tied to the issue.
- [Slack agent](/docs/guides/slack-agent): mention `@143`, DM the bot, or start from Slack App Home and keep progress tied to the conversation.
- [Autopilot](/docs/guides/autopilot): let eligible work run automatically behind visible gates.
- [Coding-agent auth](/docs/guides/coding-agent-auth): configure user and organization credentials outside prompts and repo files.
- [External API](/docs/reference/external-api): start sessions and automations from internal systems using scoped service-account tokens.
- [Self-hosting quickstart](/docs/self-hosting/single-node): bring up a production-shaped single-node instance.
- [Production deployment checklist](/docs/self-hosting/production-deployment-checklist): verify a self-hosted deployment before sending real work to agents.
