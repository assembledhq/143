# Design: PR Creation & Ship

> **Status:** Implemented | **Last reviewed:** 2026-03-25

This document describes how 143.dev creates GitHub pull requests from validated agent runs and manages the path to deployment.

## Overview

After an agent run passes validation, the system:

1. Creates a branch in the target repository
2. Commits the agent's diff
3. Opens a PR with full context
4. Monitors the PR for review status and merge
5. Detects deploys after merge

## GitHub Integration

### Authentication

143.dev connects to GitHub as a GitHub App. This provides:

- Fine-grained repository permissions
- No personal access tokens to manage
- Webhook delivery for PR events and deploys

Required permissions:
- **Contents**: Read & Write (create branches, push commits)
- **Pull Requests**: Read & Write (create/update PRs)
- **Checks**: Read (CI status)
- **Deployments**: Read (deploy detection)

The GitHub App credentials (App ID, private key) are stored in the `integrations` table.

### Installation Token

For each API call, the system generates an installation token:

```go
func (g *GitHubService) getInstallationToken(ctx context.Context, org *models.Organization) (string, error) {
    integration, _ := g.db.GetIntegration(ctx, org.ID, "github")
    // Use App private key to generate JWT, then exchange for installation token
    // Cache token until expiry
}
```

## Branch Creation

Each agent run gets a dedicated branch:

```
143/fix/{issue-id-short}/{slug}
```

Example: `143/fix/abc123/fix-null-pointer-in-user-api`

The branch is created from the repo's default branch (usually `main`):

```go
func (g *GitHubService) CreateBranch(ctx context.Context, repo string, branchName string) error {
    // 1. Get the SHA of the default branch HEAD
    ref, _ := g.client.Git.GetRef(ctx, owner, name, "refs/heads/main")

    // 2. Create the new branch
    g.client.Git.CreateRef(ctx, owner, name, &github.Reference{
        Ref:    github.String("refs/heads/" + branchName),
        Object: &github.GitObject{SHA: ref.Object.SHA},
    })
}
```

## Commit

The agent's diff is committed to the branch:

```go

    // This avoids needing to clone the repo on the server side

    // 1. Create blobs for each modified file
    // 2. Create a tree with the new blobs
    // 3. Create a commit on the tree
    // 4. Update the branch ref to point to the new commit

```

Commit message format:

```
fix: {issue.title}

Fixes #{linear_issue_id} (if Linear)
Resolves {sentry_issue_url} (if Sentry)

Generated by 143.dev agent run {agent_run.id}
```

## PR Creation

### PR Title

Format depends on the source:

- **Linear issue**: `{LINEAR-ID}: {issue.title}` (e.g., `ENG-1234: Fix null pointer in user API`)
- **Sentry issue**: `fix: {issue.title}`
- **Support ticket**: `fix: {issue.title}`

### PR Body

Uses the repo's existing PR template (`.github/pull_request_template.md`) if one exists. The system preserves the template structure, fills in its existing sections where possible, and appends only a small 143.dev/session links footer outside the template.

If no template exists, uses a default body:

```markdown
## Summary

{agent_run.result_summary}

## Issue

- **Source**: {issue.source}
- **Severity**: {issue.severity}
- **Affected customers**: {issue.affected_customer_count}
- **Occurrences**: {issue.occurrence_count}

{source-specific link: Sentry URL, Linear URL, or support ticket link}

## Changes

{high-level description of what the diff does}

## Validation

| Check | Result |
|-------|--------|
| Direction alignment | {pass/fail} |
| Correctness | {pass/fail} |
| Code quality | {pass/fail} |
| Test coverage | {pass/fail} |
| CI/CD | {pass/fail} |

## Agent Details

- **Agent**: {agent_type}
- **Token usage**: {input_tokens} in / {output_tokens} out
- **Run time**: {duration}

[143.dev](https://143.dev) | [session {run.id-short}]({dashboard_url})
```

### PR Labels

The system adds labels to the PR:

- `143-generated` — always
- `severity:{level}` — from the issue
- `source:{source}` — sentry, linear, or support

### PR Creation Code

```go
func (g *GitHubService) CreatePR(ctx context.Context, run *models.AgentRun, validation *models.Validation) (*models.PullRequest, error) {
    title := g.formatPRTitle(run)
    body := g.formatPRBody(run, validation)

    pr, _, err := g.client.PullRequests.Create(ctx, owner, repo, &github.NewPullRequest{
        Title: &title,
        Body:  &body,
        Head:  &branchName,
        Base:  github.String("main"),
    })
    if err != nil {
        return nil, err
    }

    // Add labels
    g.client.Issues.AddLabelsToIssue(ctx, owner, repo, pr.GetNumber(), labels)

    // Store PR record
    return g.db.CreatePullRequest(ctx, &models.PullRequest{
        AgentRunID:     run.ID,
        OrgID:          run.OrgID,
        GitHubPRNumber: pr.GetNumber(),
        GitHubPRURL:    pr.GetHTMLURL(),
        GitHubRepo:     fmt.Sprintf("%s/%s", owner, repo),
        Title:          title,
        Body:           body,
        Status:         "open",
        ReviewStatus:   "pending",
    })
}
```

## Review Management

### Review Assignment

Based on org settings:

- **Default**: No specific reviewers assigned (the repo's CODEOWNERS or default reviewers handle it)
- **Team assignment**: Request review from a configured GitHub team
- **Individual assignment**: Request review from specific users (round-robin or based on file ownership)

### Notifications

Configurable per org:

- **None**: PR is opened silently, reviewers find it via GitHub
- **Slack**: Post to a configured Slack channel with PR link and summary
- **Email**: Send a summary email (future)

### PR Status Tracking

The system monitors PRs via GitHub webhooks:

- `pull_request` event (action: `closed`, `merged`, `review_requested`)
- `pull_request_review` event (action: `submitted` — approved, changes_requested)
- `pull_request_review_comment` event — inline diff comments

Status updates are reflected in the `pull_requests` table and the 143.dev dashboard.

When a review is submitted on an in-scope PR (143-generated PRs by default; configurable to all PRs), the system triggers the **Review Feedback Loop** (see [11-review-feedback-loop.md](../backlog/11-review-feedback-loop.md)). Comments are run through a multi-stage filtering pipeline before LLM classification. For 143-generated PRs with `changes_requested`, the system can automatically re-run the agent with the feedback incorporated — pushing new commits to the existing PR branch instead of requiring the human to fix it. Generalizable patterns from all in-scope reviews are accumulated into a `.143/learned-conventions.md` file in the repo.

## Deploy Detection

After a PR is merged, the system detects the deploy:

### Method 1: GitHub Deployments API

Listen for `deployment_status` webhook events:

```go
func (g *GitHubService) HandleDeploymentStatus(ctx context.Context, event *github.DeploymentStatusEvent) error {
    if event.DeploymentStatus.GetState() != "success" {
        return nil
    }
    // Find the PR that introduced this commit
    pr, _ := g.findPRByCommit(ctx, event.Deployment.GetSHA())
    if pr == nil {
        return nil // not a 143.dev PR
    }
    // Record the deploy
    g.db.CreateDeploy(ctx, &models.Deploy{
        PullRequestID: pr.ID,
        Environment:   event.Deployment.GetEnvironment(),
        DeployedAt:    event.DeploymentStatus.GetCreatedAt().Time,
        CommitSHA:     event.Deployment.GetSHA(),
    })
    // Start the observation experiment
    g.jobs.Enqueue(ctx, "evaluate_experiment", ...)
}
```

### Method 2: Merge-based (Simple)

For repos without deployment tracking, treat PR merge as the deploy event. The system creates a deploy record when the PR is merged and the default branch CI passes.

### Method 3: Custom Webhook

Orgs can send a webhook to `POST /api/v1/webhooks/deploy` with commit SHA and environment to signal a deploy.

## Post-PR Actions

After a PR is created:

1. Update the `issue` status to `in_progress`
2. Update the `agent_run` to link to the PR
3. Log the PR creation in the audit trail
4. Send notifications (if configured)

After a PR is merged:

1. Update the `pull_request` status to `merged`
2. Update the `issue` status to `fixed`
3. Record the deploy
4. Enqueue the experiment evaluation job (Step 6 in the overall flow)
