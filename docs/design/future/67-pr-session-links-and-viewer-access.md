# Design: PR Session Links and Viewer Access

## Summary

143 should treat the session that produced a pull request as a first-class part of the PR's published context.

Recommended shape:

1. Keep a durable backlink from the GitHub PR to the originating 143 session.
2. Promote that backlink from a buried footer to an explicit `View 143 session` affordance in the PR body.
3. Route viewers through normal 143 auth and org-membership checks before showing the session.
4. Do **not** introduce anonymous public session transcripts in v1.

This solves the operator need behind VIR-33: when someone encounters a 143-generated PR in GitHub, they should be able to open the exact session that produced it and inspect the transcript, validation, diff, and repair history.

## Current baseline

The system already has most of the data linkage:

- `pull_requests.session_id` links each PR to exactly one session.
- Session detail already links outward to GitHub via `View PR`.
- PR creation already appends a footer link back to the session in the PR body.

Current implementation example:

```markdown
---
*Generated by [143.dev](https://143.dev) — [session abcdef01](https://app.143.dev/sessions/<session-id>)*
```

Relevant code:

- `internal/db/pull_requests.go`
- `internal/services/github/pr.go`
- `frontend/src/app/(dashboard)/sessions/[id]/session-detail-content.tsx`

## Problem

The existing footer link is not sufficient for the intended product behavior.

### 1. The backlink exists, but it is easy to miss

It is rendered as a small trailing footer after the PR body content. Reviewers scanning GitHub for "what happened here?" are unlikely to treat that as a primary navigation control.

### 2. The URL is hardcoded to `app.143.dev`

PR generation currently emits `https://app.143.dev/sessions/<id>` directly. That is incorrect for self-hosted deployments and brittle even for hosted environments.

### 3. "Other people can view the session" is mostly an access problem, not a data-link problem

The main missing behavior is not database linkage. The linkage already exists. The real product questions are:

- what happens if the PR viewer is already signed in to 143?
- what happens if they are signed out?
- what happens if they have no membership in the org that owns the session?
- what happens if they are a GitHub reviewer but not yet a 143 user?

### 4. Public anonymous sharing is riskier than it sounds

Session transcripts can contain:

- code context
- logs
- validation details
- linked issue context
- potentially sensitive prompts or attachments

That makes "anyone with the GitHub PR can read the full session" too broad as a default.

## Recommendation

### Product decision

Ship this in two layers:

1. **Durable PR backlink cleanup**: make the PR always point at the correct session URL for the deployment.
2. **Viewer access flow**: let authenticated org members open the session from GitHub with a clean redirect-through-login path.

Do not make sessions public-by-link in v1.

### Why this is the right v1

It satisfies the practical PM goal:

- teammates reviewing the PR can get to the session
- existing 143 users are not blocked
- invited teammates can sign in and land on the session

And it avoids a much larger security/design project around redaction, expiring public links, and transcript-safe rendering.

## UX

### PR body

Instead of only a small footer sentence, the PR should contain a clearly labeled block near the bottom of the generated metadata:

```markdown
## 143 Session

View the full 143 session for transcript, validation, preview, and repair actions:

[Open session in 143](https://<app-base-url>/sessions/<session-id>)
```

Keep the lightweight generated-by footer if desired, but the explicit block should be the primary affordance.

### Signed-in viewer

If the viewer is already signed in to 143 and belongs to the owning org:

- open the session detail page directly

### Signed-out viewer

If the viewer is not signed in:

- send them through login
- preserve the target session URL
- return them to that session after auth completes

### Viewer without org membership

If the viewer is authenticated but does not belong to the owning org:

- show a clear "You do not have access to this session" state
- offer the org/invite flow if the product has enough identity context to do so

Do not silently dump them on a generic dashboard home page. The redirect should preserve intent.

## Non-goals

- Anonymous public session transcripts
- GitHub-side embedded transcript rendering
- Cross-org visibility bypasses
- Session sharing tokens separate from normal auth

Those may become future work, but they are not required to solve VIR-33 well.

## Implementation notes

### 1. Use a deployment-configured app base URL

The PR body generator should stop hardcoding `https://app.143.dev`.

Recommended shape:

- give the GitHub PR service the same concept Linear already uses for session URLs: a configured app base URL
- build `https://<app-base-url>/sessions/<id>` from that config

This should be used consistently in:

- default PR body generation
- LLM/template footer appends
- any future PR comment or repair-link surfaces

### 2. Treat the PR body as the canonical outbound surface

The simplest stable place for the backlink is still the PR body because it:

- persists with the PR
- is visible to all reviewers
- needs no extra webhook choreography

No extra database work is needed for this. The authoritative link already comes from `pull_requests.session_id`.

### 3. Preserve auth-return context

If a GitHub reviewer clicks the link while signed out, auth should return them to `/sessions/<id>` after login rather than dropping them onto a generic home screen.

### 4. Keep session access governed by org membership

The PR link should not bypass the existing product security model. The PR is a discoverability surface, not an authorization grant.

## Open questions

1. Should the PR body block say `View 143 session` or `View build session`? `143 session` is accurate; `build session` may scan better for non-users.
2. Should the app offer an invite-request CTA when a signed-in viewer lacks org membership?
3. Should the link target the default session overview, or a focused deep link like `?tab=overview` or `?view=changes`?

## PM conclusion

VIR-33 is best understood as a **productization and access-flow task**, not a new relational-model task.

The PR-to-session relation already exists today. The missing work is:

- make the link deployment-correct
- make it visually explicit in the PR
- make the auth and no-access flows preserve reviewer intent

That is the smallest change set that actually makes "other people can view the 143 session from the PR" true in practice.
