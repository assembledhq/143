# Design: Failure Communication

When an agent run fails, the user experience matters as much as the technical classification. This document designs how 143.dev communicates failures to users in a way that is honest, helpful, and not demoralizing.

## Problem

Today, when an agent fails, the user sees a status badge ("Failed") and maybe an error string ("agent exited with code 1"). This tells the user nothing useful:

- Why did it fail?
- Is this a bug the agent can't handle, or was it a transient error?
- What should the user do next?
- Should they try again, provide more context, or fix it manually?

Bad failure communication erodes trust in the system. After 2-3 opaque failures, users stop clicking "Fix This."

## Design: Human-Readable Failure Explanations

Every failed run gets a structured failure summary written in plain English, generated by an LLM post-processing step. This is stored on the `agent_runs` record and displayed prominently in the UI.

### Failure Analysis

When a run fails (agent error, validation failure, or timeout), a background job runs a lightweight LLM analysis:

```go
type FailureSummary struct {
    Explanation  string   // 1-3 sentence human-readable explanation
    Category     string   // "context", "complexity", "tooling", "validation"
    NextSteps    []string // 2-3 actionable suggestions
    RetryAdvised bool     // should the user try again?
}

func (s *FailureService) AnalyzeFailure(ctx context.Context, run *models.AgentRun) (*FailureSummary, error) {
    prompt := fmt.Sprintf(`An AI coding agent attempted to fix this issue and failed.

Issue: %s
Issue description: %s
Agent error: %s
Agent output (last 500 chars): %s
Validation result: %s

Write a 1-3 sentence explanation for a software engineer about why this fix attempt failed.
Then suggest 2-3 concrete next steps they could take.
Finally, indicate whether retrying the same run is likely to help (true/false).

Be honest and specific. Do not use vague language like "an error occurred."
If the issue is genuinely too complex for automated fixing, say so.`,
        run.Issue.Title, run.Issue.Description, run.Error, run.LastOutput, run.ValidationDetails)

    return s.llm.Analyze(ctx, prompt)
}
```

### Failure Categories

Keep categories simple. Four buckets, each with a distinct user-facing message pattern:

| Category | What Happened | Example Explanation |
|----------|--------------|---------------------|
| `context` | Agent couldn't understand the codebase well enough | "The agent couldn't find the source of this error. The stack trace references `legacy/v2/handler.go`, which doesn't exist in the current codebase. This error may be from a previous version." |
| `complexity` | Issue is too complex for automated fixing | "This issue involves changes across 12 files spanning 3 services. The agent attempted a fix but it only addressed part of the problem. This issue likely needs a human to make architectural decisions." |
| `tooling` | Infrastructure problem (timeout, sandbox crash) | "The agent ran out of time (5 minute limit) while still analyzing the codebase. This might succeed with a longer timeout, or the codebase may need better documentation to help the agent navigate faster." |
| `validation` | Agent produced a fix but it didn't pass checks | "The agent wrote a fix, but it broke 3 existing tests in `users_test.go`. The fix changed how nil profiles are handled, but existing tests expect the old behavior. The fix approach may need to be more conservative." |

### Next Steps

Each failure category maps to a default set of actionable suggestions:

| Category | Suggested Next Steps |
|----------|---------------------|
| `context` | 1. "Add context about this area to your CLAUDE.md or .143/context.md" 2. "Check if the stack trace references outdated file paths" 3. "Try providing a more detailed issue description" |
| `complexity` | 1. "Break this into smaller sub-issues that the agent can tackle individually" 2. "Use the agent to investigate the root cause, then fix it yourself" 3. "Add more test coverage to the affected area first" |
| `tooling` | 1. "Retry the run — this may be a transient issue" 2. "Increase the timeout in agent settings" 3. "Check that your repo builds cleanly in a fresh environment" |
| `validation` | 1. "Review the agent's diff — it may be close to correct" 2. "Retry with additional context about the expected behavior" 3. "Check if the failing tests are testing the right behavior" |

## Failure Rate Transparency

### Setting Expectations

On the dashboard and in onboarding, display the system's current fix rate transparently:

```
+----------------------------------------------------------+
|  143.dev Fix Rate                                        |
|                                                          |
|  Overall: 42% of attempted issues get a merged fix       |
|                                                          |
|  By type:                                                |
|    Null/nil errors:     68% success rate                 |
|    Type errors:         55% success rate                 |
|    Logic bugs:          28% success rate                 |
|    Performance issues:  15% success rate                 |
|                                                          |
|  Your repo (last 30 days):                               |
|    8 attempted, 4 merged, 2 in review, 2 failed          |
+----------------------------------------------------------+
```

This sets honest expectations. Users who know the fix rate is 42% won't be demoralized by individual failures. Users who see their repo's rate is below average get a signal that they should improve their codebase context.

### Fix Rate Computation

```go
func (s *AnalyticsService) ComputeFixRate(ctx context.Context, orgID uuid.UUID, window time.Duration) (*FixRateMetrics, error) {
    runs, _ := s.db.GetCompletedRuns(ctx, orgID, time.Now().Add(-window), time.Now())

    total := len(runs)
    merged := 0
    for _, r := range runs {
        if r.PR != nil && r.PR.Status == "merged" {
            merged++
        }
    }

    return &FixRateMetrics{
        Total:      total,
        Merged:     merged,
        FixRate:    float64(merged) / float64(total),
    }, nil
}
```

When a failure is sent via Slack or email, include the explanation and next steps inline:


## Database Changes

### Updated: `agent_runs` table

Add columns for failure communication:

| New Column | Type | Notes |
|-----------|------|-------|
| failure_explanation | text | Human-readable 1-3 sentence explanation (null if not failed) |
| failure_category | text | `context`, `complexity`, `tooling`, `validation` (null if not failed) |
| failure_next_steps | text[] | Actionable suggestions for the user |
| failure_retry_advised | boolean | Whether retrying is likely to help |

### Job Queue

| Job Type | Queue | Trigger |
|----------|-------|---------|
| `analyze_failure` | `system` | After a run ends in `failed` or validation fails |

## Build Order

This is part of **Phase 4** (Agent Execution). Failure communication should ship alongside the first agent runs, not as a later addition:

1. **Failure analysis job** — LLM-based post-failure analysis, store on `agent_runs`
2. **Failure UI** — explanation card on run detail page, retry/dismiss actions
3. **Fix rate dashboard** — aggregate fix rates, per-type breakdown, per-repo stats
4. **Notification integration** — include failure explanations in Slack/email notifications
